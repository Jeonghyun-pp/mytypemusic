## 2. 기술 평가

### 평가 대상 6개 기술

#### Ingestion (데이터 수집 패턴)

- **정의**: 외부 소스에서 데이터를 수집 → 정규화 → 중앙 저장소로 이동시키는 프로세스
- **현재 적용 상태**: 이미 적용됨 (`src/lib/trends/`의 5개 소스 → `TrendItem` 정규화 → `TrendSnapshot` DB 저장)

#### Promtail

- **정의**: 로그 파일을 수집해서 Loki로 전송하는 경량 에이전트 (Grafana 스택)
- **동작**: 로그 파일/systemd journal을 tail → 메타데이터 라벨 부착 → Loki로 전송
- **상태**: 2025.02 deprecated → Grafana Alloy로 대체, 2026.03 EOL

#### Grafana Loki

- **정의**: "Prometheus와 같지만 로그용" 로그 집계 시스템
- **특징**: 로그 내용이 아닌 라벨만 인덱싱 (저비용), LogQL 쿼리, S3/GCS에 압축 저장
- **통합**: Grafana + Prometheus + Tempo와 연동하여 완전한 관측성 스택

#### Apache Kafka

- **정의**: 분산 이벤트 스트리밍 플랫폼
- **특징**: 초당 수백만 이벤트, 디스크 영속 저장, 재처리 가능, 소비자 그룹으로 다중 서비스 독립 소비
- **용도**: 이벤트 소싱, CDC, 로그 집계, 실시간 분석, 서비스 간 디커플링

#### Prometheus

- **정의**: 메트릭 수집/저장/알림 시계열 데이터베이스
- **특징**: Pull 기반 (타겟의 /metrics를 주기적 scrape), PromQL 쿼리, 다차원 라벨
- **용도**: 인프라 모니터링, SLA 추적, 알림

#### Apache Airflow

- **정의**: Python DAG 기반 워크플로우 오케스트레이터
- **특징**: Cron + 이벤트 트리거, UI 대시보드, HITL 지원 (3.x), 5,800+ 기업 사용
- **최신**: 3.0 (2025.04) → 3.15 (현재), DAG 버전 관리, 이벤트 드리븐 스케줄링

### 자체 사용 기준 평가

| 기술 | 적합성 | 이유 |
|------|--------|------|
| Ingestion | ✅ 이미 적용됨 | `trends/` 시스템이 이 패턴 |
| Promtail | ❌ 불필요 | Deprecated + 서버리스 비호환 |
| Loki | ⚠️ Grafana Cloud로 | LLM 호출 로그 중앙화에 유용 |
| Kafka | ❌ 오버킬 | 하루 수십~수백 건 처리에 분산 스트리밍 불필요 |
| Prometheus | ⚠️ Grafana Cloud로 | Pull 모델이 서버리스와 비호환 |
| Airflow | ❌ 비추 | Python 기반 + 상시 서버 필요 + TS 코드와 브릿지 비용 |

### 자체 사용 시 추천 대안

| 대안 | 장점 | 용도 |
|------|------|------|
| **Inngest** | Vercel 네이티브, TS 네이티브, `npm install` 하나로 시작, 무료 25K step/월 | Job Queue + Cron 대체 |
| **Trigger.dev** | 장시간 실행 지원, 오픈소스 self-host 가능 | Inngest 대안 |
| **Temporal** | TS SDK, 가장 강력한 워크플로우 엔진 | 규모 확장 시 |
| **BullMQ** | TS 네이티브, Redis 기반, 현재 큐 업그레이드 | Job Queue 강화 |
| **Grafana Cloud** | 무료 50GB 로그 + 10K 메트릭, 서버 운영 불필요 | 관측성 |
| **Sentry** | 에러 추적 + 트랜잭션 모니터링 | 에러 추적 |

## 3. SaaS 판매 시 아키텍처

서비스를 SaaS로 판매할 경우, 멀티테넌트 요구사항으로 인해 기술 평가가 완전히 달라짐.

### 규모 변화 시뮬레이션

| 항목 | 자체 사용 | SaaS 10고객 | SaaS 100고객 | SaaS 1000고객 |
|------|----------|------------|-------------|--------------|
| 일간 트렌드 수집 | 수십 회 | 수백 회 | 수천 회 | 수만 회 |
| 일간 LLM 호출 | 수십 회 | 수백 회 | 수천 회 | 수만 회 |
| 파이프라인 동시 실행 | 1~2 | 10~20 | 50~100 | 200+ |
| Autopilot 제안/일 | 수 건 | 수십 건 | 수백 건 | 수천 건 |
| 로그량/일 | MB | 수백 MB | GB | 수십 GB |

### SaaS 기준 기술 재평가

| 기술 | 자체용 | SaaS | SaaS에서 필요한 이유 |
|------|--------|------|---------------------|
| Kafka | ❌ | ✅ 100+고객부터 | 고객 격리, LLM 장애 시 재처리, 과금 이벤트 추적 |
| Airflow | ❌ | ✅ 200+고객부터 | 배치 과금 집계, 데이터 정리, 리포트 생성 |
| Prometheus | ⚠️ | ✅ 필수 | SLA 보장, 용량 계획, 비용 알림 |
| Loki | ⚠️ | ✅ 필수 | 고객별 디버깅, LLM 비용 감사, 고객 지원 |
| Alloy | ❌ | ✅ K8s 전환 시 | 컨테이너 로그 자동 수집 |

### SaaS에서 Kafka 활용

```
┌─ Kafka Topics ─────────────────────────────────────────┐
│  pipeline.requests    → 파이프라인 요청 큐               │
│  trend.fetch          → 트렌드 수집 이벤트               │
│  llm.calls            → LLM 호출 큐 + 비용 추적         │
│  publish.scheduled    → SNS 발행 스케줄                 │
│  analytics.events     → 성과 데이터 수집                │
│  billing.usage        → 고객별 사용량 과금               │
└─────────────────────────────────────────────────────────┘

핵심 가치:
  • 내구성: OpenAI 장애 30분 → 수백 건 요청 보존 → 복구 후 자동 재처리
  • 격리: 고객 A 폭주가 고객 B에 영향 없음 (파티션 기반)
  • 과금: billing.usage 토픽으로 정확한 종량제 과금
  • 스케일: Consumer Group 수평 확장
```

### SaaS에서 Airflow DAG 설계

```
DAG: daily_trend_collection (@hourly)
  → 전체 고객 키워드를 배치로 묶어 효율적 수집
  → 소스별 API 할당량 관리 (YouTube 10K/일 → 고객별 분배)

DAG: autopilot_batch_scan (*/30 * * * *)
  → 활성 고객별 Autopilot 설정 순회
  → 우선순위 기반 처리 (유료 고객 우선)

DAG: billing_aggregation (@daily)
  → 고객별 LLM 토큰 사용량, API 호출 수, 스토리지 집계
  → 인보이스 생성

DAG: analytics_rollup (@daily)
  → 고객별 성과 데이터 집계 → 리포트 생성 + 이메일 발송

DAG: cleanup_expired_data (@weekly)
  → 해지 고객 데이터 정리, 오래된 TrendSnapshot 아카이빙
```

### SaaS에서 관측성 (Prometheus + Loki + Grafana)

```
Prometheus 메트릭:
  비즈니스  → active_customers_total, revenue_mrr_usd
  SLA      → pipeline_latency_p99_seconds, api_availability_ratio
  용량     → llm_queue_depth, worker_utilization_ratio, db_connection_pool_usage
  비용     → llm_cost_usd{customer, model}
  알림     → "P99 > 5분" → PagerDuty, "에러율 > 10%" → Slack

Loki 로그:
  고객 지원 → {customer="acme"} |= "pipeline" |= "error"
  LLM 감사 → 모든 LLM 호출의 프롬프트/토큰/비용 로그
  과금 분쟁 → 사용량 근거 로그 추적
```

### SaaS 전체 아키텍처

```
┌─ 고객 대면 ──────────────────────────────────────────┐
│  Next.js (Studio UI)                                  │
│  API Gateway (인증, rate limit, 멀티테넌트 라우팅)      │
└──────────────┬────────────────────────────────────────┘
               ↓
┌─ 이벤트 버스 ────────────────────────────────────────┐
│  Apache Kafka                                         │
│  pipeline.requests │ trend.fetch │ publish.schedule   │
│  llm.calls         │ billing.usage │ analytics.events │
└──────────────┬────────────────────────────────────────┘
               ↓
┌─ 워커 서비스 (스케일 아웃) ───────────────────────────┐
│  Pipeline Workers  │ Trend Collectors                 │
│  SNS Publishers    │ Billing Processor                │
└──────────────┬────────────────────────────────────────┘
               ↓
┌─ 배치 오케스트레이션 ────────────────────────────────┐
│  Apache Airflow                                       │
│  트렌드 배치 수집 │ 과금 집계 │ 리포트 │ 데이터 정리   │
└──────────────┬────────────────────────────────────────┘
               ↓
┌─ 관측성 ─────────────────────────────────────────────┐
│  Prometheus (메트릭) │ Loki (로그) │ Grafana (대시보드) │
│  Alloy (컨테이너 로그 수집)                             │
└──────────────┬────────────────────────────────────────┘
               ↓
┌─ 데이터 저장 ────────────────────────────────────────┐
│  PostgreSQL (메인 DB)  │ pgvector (RAG 임베딩)        │
│  Redis (캐시, 세션, rate limit)                       │

## 4. 단계별 도입 로드맵

### 자체 사용 (현재)

```
즉시      → Inngest 도입 (jobs/queue.ts + Vercel Cron 대체)
          → Sentry 추가 (에러 추적)
3개월 후  → Autopilot 전체를 Inngest function으로 마이그레이션
          → E2E 파이프라인도 Inngest 전환 (SSE는 별도 유지)
규모 확장 → Temporal 검토
```

### SaaS 판매 시

| 단계 | 고객 수 | 도입 기술 | 이유 |
|------|---------|----------|------|
| **MVP** | 1~10 | Inngest + Sentry + Grafana Cloud | 서버리스 유지, 최소 비용 |
| **초기 성장** | 10~50 | + Redis (BullMQ) + Prometheus (Cloud) | Job 큐 강화, 메트릭 시작 |
| **스케일** | 50~200 | + Kafka + Loki | 이벤트 격리, 로그 중앙화 |
| **엔터프라이즈** | 200+ | + Airflow + Alloy + K8s | 배치 오케스트레이션, 컨테이너 전환 |

### 핵심 원칙

SaaS 전환 시 모든 기술 도입의 근거가 되는 원칙:

> **"고객 A의 문제가 고객 B에 영향을 주면 안 된다"**

이 하나의 원칙이 Kafka(이벤트 격리), Prometheus(SLA 모니터링), Loki(고객별 디버깅), Airflow(배치 관리) 도입의 핵심 이유가 된다.

## 5. 현재 프로덕트 약점 분석

> SaaS 준비도 평가: **2/10** — 잘 만들어진 1인용 내부 도구이지만, 판매용 제품으로는 근본적 구조 변경 필요

### 5.1 CRITICAL — 서비스 판매 불가

#### 인증 & 사용자 시스템 없음

- `SITE_PASSWORD` 하나로 전체 접근 제어 (쿠키 30일)
- Prisma 스키마에 `User`, `Account`, `Organization` 테이블 없음
- 세션/JWT, OAuth, RBAC(역할 기반 접근 제어) 없음
- API 키 관리, 워크스페이스 지원 없음

#### 멀티테넌시 완전 부재

- 모든 모델에 `tenantId` 컬럼 없음
- `prisma.autopilotConfig.findMany({})` — WHERE 없이 전체 데이터 조회
- SnsAccount, Publication, WritingPersona, Job 전부 글로벌 공유
- 고객 A의 데이터가 고객 B에게 노출되는 구조

#### 결제 & 과금 시스템 없음

- Stripe, Paddle 등 결제 연동 없음
- 구독 모델, 요금제, 사용량 측정 없음
- LLM 호출 비용 추적 없음
- `Subscription`, `Plan`, `Invoice` 테이블 없음

#### 파일 저장소: 로컬 디스크 사용

- `path.resolve(process.cwd(), "../../outputs/reels/uploads")` — Vercel 재배포 시 전부 삭제
- 업로드 파일 크기 제한 없음
- S3/GCS 클라우드 스토리지 연동 없음

### 5.2 HIGH — 운영 장애 위험

#### Job 큐: Dead Letter Queue 없음

- 3회 재시도 후 실패한 Job → DB에 방치 (알림 없음)
- Autopilot 제안 생성 실패 → catch 블록에서 에러만 잡고 무시
- 댓글 수집 실패 → 플랫폼별 에러 추적 없음

#### 외부 API 장애 전파

| API | 문제 |
|-----|------|
| OpenAI | rate limit 시 `retry-after` 헤더 미사용, 하드코딩 대기 |
| YouTube | 일 10K 쿼터 추적 없음, 소진 시 조용히 `[]` 반환 |
| Google Trends | npm 스크래퍼 기반, 연 2-3회 깨짐, 대체 수단 없음 |
| Brave/Tavily | API 키 없으면 조용히 스킵 |
| 공통 | fetch 호출에 timeout 미설정 → 무한 대기 가능 |

#### 캐시: 인메모리 전용

- 서버리스 인스턴스마다 별도 캐시 → 스케일 아웃 시 불일치
- 트렌드 캐시(15분), 제안 캐시(30분), 디자인 품질 캐시(1000건) 모두 인메모리
- 서버 재시작 시 전체 소실, Redis 등 분산 캐시 없음

#### DB 쿼리: N+1 문제

- `scanner.ts` — 승인된 제안마다 config 개별 조회 + publishNow 개별 호출
- `analyticsCollect.ts` — 계정마다 4+ 쿼리, 바운드 없는 findMany
- `commentFetch.ts` — 게시물마다 댓글 조회 + upsert 루프

#### 타임아웃 불일치

| 구간 | 설정값 |
|------|--------|
| Vercel 함수 한도 | 60초 |
| Job 핸들러 타임아웃 | 55초 |
| LLM Vision API | 60초 (Vercel과 같거나 초과) |
| 트렌드 수집 | 미설정 |
| Autopilot 제안 | 미설정 |

#### 보안 취약점

| 항목 | 상태 |
|------|------|

if (usage) {
  await prisma.llmUsageLog.create({
    data: {
      model: opts?.model ?? "gpt-4o-mini",
      promptTokens: usage.prompt_tokens,
      completionTokens: usage.completion_tokens,
      costUsd: calculateCost(model, usage),  // 모델별 단가 계산

// 2. 비용 계산 헬퍼
function calculateCost(model: string, usage: Usage): number {
  const rates: Record<string, { input: number; output: number }> = {
    "gpt-4o-mini":  { input: 0.15 / 1e6,  output: 0.60 / 1e6 },
    "gpt-4o":       { input: 2.50 / 1e6,  output: 10.0 / 1e6 },
  };
  const rate = rates[model] ?? rates["gpt-4o-mini"];
  return usage.prompt_tokens * rate.input + usage.completion_tokens * rate.output;
}

// 3. Prisma 스키마 추가
model LlmUsageLog {
  id               String   @id @default(cuid())
  model            String
  promptTokens     Int
  completionTokens Int
  costUsd          Float
  caller           String   // "autopilot", "pipeline", "suggestions" 등
  createdAt        DateTime @default(now())
  @@index([createdAt])
  @@index([caller])
}
```

#### 8. Inngest 도입

**현재 상태:**

```typescript
// jobs/queue.ts — 커스텀 큐 시스템 (약 144줄)
const HANDLER_TIMEOUT_MS = 55_000;
const STUCK_JOB_THRESHOLD_MS = 10 * 60 * 1000;  // 10분

export async function processJobs(batchSize = 10) {
  // 1. stuck 상태 job 복구 (10분 이상 running)
  const stuckJobs = await prisma.job.updateMany({
    where: { status: "running", updatedAt: { lt: tenMinAgo } },
    data: { status: "pending", error: "Auto-recovered..." },
  });

  // 2. pending job 배치 조회 (priority DESC, scheduledAt ASC)
  const jobs = await prisma.job.findMany({
    where: { status: "pending", scheduledAt: { lte: now } },
    orderBy: [{ priority: "desc" }, { scheduledAt: "asc" }],
    take: batchSize,
  });

  // 3. 각 job 순차 실행
  for (const job of jobs) {
    await prisma.job.update({ data: { status: "running" } });
    try {
      await withTimeout(handler(job.payload), HANDLER_TIMEOUT_MS);
      await prisma.job.update({ data: { status: "completed" } });
    } catch {
      if (job.retryCount >= maxRetries) {
        await prisma.job.update({ data: { status: "failed" } });
      } else {
        const delay = retryCount * retryCount * 60_000;  // 1분, 4분, 9분
        await prisma.job.update({
          data: { status: "pending", scheduledAt: new Date(now + delay) },
        });
      }
    }
  }
}
// 문제:
// • Vercel Cron이 5분마다 POST /api/jobs/process 호출 → 최대 5분 지연
// • stuck 감지가 10분 → 장애 발생부터 복구까지 최대 15분
// • 재시도 로직/타임아웃/복구를 직접 유지보수
// • 실행 히스토리 UI 없음 — DB 직접 조회해야 함
```

// api/reels/upload/route.ts — 로컬 파일시스템 저장
export async function POST(req: Request) {
  const formData = await req.formData();
  const file = formData.get("file") as File;
  // ← 파일 크기 제한 없음!

  const buffer = Buffer.from(await file.arrayBuffer());

  const uploadDir = path.resolve(process.cwd(), "../../outputs/reels/uploads");
  await fs.mkdir(uploadDir, { recursive: true });
  await fs.writeFile(path.join(uploadDir, filename), buffer);

  // ← 디스크 용량 제한 없음

// api/reels/file/route.ts — 로컬 파일 서빙
export async function GET(req: Request) {
  const filename = url.searchParams.get("name");
  // 경로 순회 방지 (기본적)
  if (!filename || filename.includes("..") || filename.includes("/") || filename.includes("\\")) {
    return new Response("Invalid", { status: 400 });
  }
  const filePath = path.join(uploadDir, filename);
  const buffer = await fs.readFile(filePath);
  // ← 파일이 Vercel 재배포로 삭제되었으면 500 에러
  // ← Content-Length 체크 없이 전체 파일 메모리 로드 → 대용량 파일 시 OOM
}

// 1. 스토리지 헬퍼 (lib/storage.ts)
import { S3Client, PutObjectCommand, GetObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";

const s3 = new S3Client({
  region: "auto",
  endpoint: process.env.R2_ENDPOINT,       // https://<account>.r2.cloudflarestorage.com
  credentials: {
    accessKeyId: process.env.R2_ACCESS_KEY!,
    secretAccessKey: process.env.R2_SECRET_KEY!,
  },
});

export async function uploadFile(key: string, buffer: Buffer, contentType: string) {
  await s3.send(new PutObjectCommand({
    Bucket: process.env.R2_BUCKET!,
    Key: key,
    Body: buffer,
    ContentType: contentType,
  }));
  return `${process.env.R2_PUBLIC_URL}/${key}`;
}

export async function getSignedDownloadUrl(key: string) {
  return getSignedUrl(s3, new GetObjectCommand({
    Bucket: process.env.R2_BUCKET!,
    Key: key,
  }), { expiresIn: 3600 });
}

// 2. upload route 수정
const MAX_FILE_SIZE = 50 * 1024 * 1024;  // 50MB 제한 추가

export async function POST(req: Request) {
  const file = formData.get("file") as File;
  if (file.size > MAX_FILE_SIZE) {
    return NextResponse.json({ error: "File too large" }, { status: 413 });
  }
  const buffer = Buffer.from(await file.arrayBuffer());
  const key = `reels/${Date.now()}-${crypto.randomUUID()}${ext}`;
  const url = await uploadFile(key, buffer, file.type);
  return NextResponse.json({ url });
}

// 3. file route → signed URL 리다이렉트
export async function GET(req: Request) {
  const key = url.searchParams.get("key");
  const signedUrl = await getSignedDownloadUrl(`reels/${key}`);
  return Response.redirect(signedUrl);
}
